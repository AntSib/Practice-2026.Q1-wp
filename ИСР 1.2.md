# Архитектура решения

Разработанный сценарий CI реализован с использованием YAML-конфигурации системы непрерывной интеграции веб-приложения на основе фреймворка Django и Next.js для платформы **Sourcecraft**.


## Логика работы CI-сценария

CI-конфигурация состоит из трёх основных компонентов:

* триггер запуска
* workflow
* задача выполнения сборки

Пайплайн выполняет следующие операции:

1. Отслеживание изменений в основной ветке репозитория.
2. Запуск workflow сборки контейнера.
3. Инициализация контейнера со средой Docker.
4. Аутентификация в контейнерном реестре Yandex Cloud.
5. Сборка Docker-образа приложения.
6. Назначение тегов образу.
7. Публикация образа в удалённый реестр.


### Триггер запуска пайплайна

Пайплайн запускается при выполнении события `push` в ветку `main`.

Благодаря этому происходит запуск сборки только для основной ветки проекта и гарантируется публикацию только стабильной версии приложения.


### Workflow сборки

В конфигурации определён workflow `build-workflow`, который описывает общий процесс выполнения CI-задачи.

Workflow включает одну задачу (`task`) — `build-task`.

Эта задача отвечает за сборку Docker-образа и его публикацию.


### Среда выполнения

Для выполнения сборки используется контейнерная среда:

```bash
image: docker:24.0
```

для наличия Docker CLI, позволяющей:
* выполнять команды `docker build` и `docker push`;
* исключить необходимость ручной установки Docker в раннере.


# Механизм работы задачи сборки

Основная логика работы реализована внутри блока `script`.

Последовательность действий следующая.


## Проверка доступности Docker

```bash
docker version
```

Данная команда используется для диагностики среды выполнения и подтверждения доступности Docker-daemon внутри CI-контейнера.


## Аутентификация в контейнерном реестре

Для доступа к **Yandex Container Registry** используется OAuth-токен.

Авторизация выполняется командой:

```bash
docker login cr.yandex
```

Передача токена осуществляется через стандартный поток ввода (`stdin`):

```bash
echo "$YC_OAUTH_TOKEN" | docker login ...
```

Использование `--password-stdin` обеспечивает:

* безопасность передачи токена;
* исключение его отображения в логах;
* соответствие рекомендациям Docker по работе с секретами.

OAuth-токен хранится в CI-секретах и не присутствует в исходном коде репозитория.

## Формирование тегов образа

Перед сборкой формируется дополнительный тег на основе идентификатора коммита:

```bash
IMAGE_TAG_SHORT=${CI_COMMIT_SHA:0:7}
```

В результате образ получает два тега:

* `latest` — последняя версия приложения
* `<commit_sha>` — версия, связанная с конкретным коммитом


## Формирование URI образа

URI контейнерного образа формируется динамически:

```bash
cr.yandex/<registry-id>/<image-name>
```

Значения `registry-id` и `image-name` передаются через переменные окружения CI.

## Сборка Docker-образа

Сборка выполняется командой:

```bash
docker build
```

В процессе сборки образу назначаются два тега:

```bash
latest
commit_sha
```

Это достигается передачей нескольких параметров `-t`.

Такой подход позволяет избежать повторной сборки для каждого тега.


## Публикация образа

После успешной сборки выполняется публикация образа в удалённый реестр:

```bash
docker push
```

Отправляются оба тега:

* `latest`
* тег коммита

В результате образ становится доступным для использования при развёртывании приложений.


Применение двух тегов (`latest` и commit-hash) обеспечивает:

* наличие текущей версии образа;
* возможность точного воспроизведения конкретной сборки.
